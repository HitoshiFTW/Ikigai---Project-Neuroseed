## I. Territorial Perception Layer

The first architectural phase instantiated structural self-awareness. We implemented three tightly coupled classes -- `StructureMapBuilder`, `DependencyLinkMemory`, and `SemanticRegionGuard` -- forming the organism's first internal territorial perception layer.

`StructureMapBuilder` constructs a live symbolic map of structural regions from the active workspace and artifact signals each waking tick. It organizes the artifact's internal topology into a strict four-layer semantic region hierarchy: epoch banners at depth 1 marking temporal development phases (DAY_14, DAY_22, DAY_24, DAY_25 and all variants); subsystem cluster regions at depth 2 grouping functionally related class assemblies such as sleep circuitry, memory surfaces, and cognition blocks; class territory regions at depth 3 representing individual class boundaries and their exported symbols; and micro-regions at depth 4 covering fine-grained structural elements such as serialization surfaces, test validation blocks, and rollback ancestry hooks. Each region carries a `confidence` score initialized at `0.5` and reinforced by `+0.10` per matching tick, capped at `1.0`, implementing the organism's first region identity persistence mechanism. The builder maintains a persistent region pool (`_region_pool`) across ticks, ensuring that re-encountered structural regions accumulate confidence rather than being reconstructed from scratch. Protected biology surfaces -- `HomeostasisSystem`, `Neuron`, `Synapse`, `HPAAxisSystem`, `CortisolSystem`, `AdenosineSystem`, and their registered kin -- are flagged with `protected: True` and tagged with `invariants: ['protected_biology']`, rendering them permanently ineligible for ordinary edit approval.

`DependencyLinkMemory` tracks directed structural dependency relationships and computes the blast radius of any proposed edit. It reads the source region from the current structure map and resolves the downstream impact surface using the `_BLAST_WEIGHTS` table -- a frozen mapping from class name to base blast impact, where `HomeostasisSystem` anchors at `0.95`, `Neuron` and `Synapse` at `0.90`, and cognitive-layer classes such as `VersionDiffTracker` and `NarrativeMemory` below `0.50`. Blast radius is further amplified by `+0.10` when a rollback or drift event is active, enforcing the conservative principle that unstable structural states magnify downstream risk. Edge weights are persistent across ticks and form the first internal dependency graph memory.

`SemanticRegionGuard` serves as the final gatekeeping layer before any edit intent propagates downstream to `PatchPreviewMemory`. It inspects the live structure map, dependency blast radius, and integrity state in sequence to produce one of four guard states: `protected_zone_violation` when a protected biology surface is implicated and blast radius exceeds `0.75`; `serialization_drift` when an unsafe patch targets a serialization micro-region; `boundary_violation` when blast radius is elevated without a biology component; and `safe_region_patch` when structural conditions permit progression. Critically, `SemanticRegionGuard` introduces zero drift to the Day 24 pipeline: it inserts before edit intent reaches `PatchPreviewMemory`, ensuring that no structurally inadmissible edit ever enters the patch preview or execution chain.

All three classes operate exclusively in the waking branch with no neuromodulator interactions, no drive writes, and no motor loop hooks. The five required module-level constants -- `_PROTECTED_CLASS_NAMES`, `_EPOCH_PREFIXES`, `_CLUSTER_KEYWORDS`, `_BLAST_WEIGHTS`, and `_SERIALIZATION_KEYS` -- are frozen at the ikigai.py module level, serializable through standard to_dict/from_dict round-trips.

## II. Ordered Rewrite Migration Layer

With territorial perception active, the second architectural phase introduced procedural rewrite cognition. We implemented `RegionRewritePlanner`, `InterfaceStabilityChecker`, and `CrossRegionCommitGraph` -- forming the organism's first dependency-ordered rewrite migration layer.

`RegionRewritePlanner` converts the structure map, dependency blast surface, and edit intent signal into an ordered symbolic rewrite plan. The plan is always generated in dependency-first sequence: the source region (the edit origin) occupies step 1; any detected downstream region occupies step 2; the serialization surface occupies step 3 as an update target; test validation occupies the penultimate step; and rollback ancestry update always occupies the final step. This ordering law is invariant -- serialization is always updated before rollback is confirmed, and tests are always validated before ancestry is committed. Each step carries a `risk_score`, a `requires_guard_escalation` flag, and an operation name drawn from the `_REWRITE_OPS` constant set. Protected biology regions -- specifically those whose region identifiers contain stems of registered biology class names -- receive `requires_guard_escalation: True` and elevated risk scores, forcing any downstream consumer to handle them with explicit guard review. When a plan step involves a protected class, `requires_escalation` at the plan record level is set `True`, providing a single-field signal sufficient for any downstream rejection gate.

`InterfaceStabilityChecker` validates whether the proposed rewrite plan preserves interface contracts. It reads the plan steps, the dependency blast surface, and the rollback controller state to detect four failure modes: `migration_required` when a `migrate_interface` operation is present in the plan steps -- indicating that at least one class public contract will change; `serialization_break_risk` when a serialization micro-region is being modified without a corresponding `update_serialization` step -- a structural inconsistency that would silently corrupt downstream to_dict/from_dict round-trips; `rollback_break_risk` when rollback is active and blast radius is high, indicating that the rewrite would compromise the rollback surface; and `interface_stable` when none of these conditions hold. The stability score is `plan_confidence * 0.65 + blast_confidence * 0.35`. Migration region lists are recorded per check, providing an explicit list of interface-breaking classes for any downstream review gate.

`CrossRegionCommitGraph` groups successful plan-stability pairs into ancestry-tracked commit records. Each commit carries a unique `commit_id`, a `region_chain` list, a `success_score`, a `rollback_parent` pointing to the most recent stable ancestor (success score >= 0.60), and a `partial_success` flag with a `failed_regions` list identifying which high-risk steps did not complete cleanly (risk above 0.70 threshold). The EMA reinforcement law governs the `_chain_scores` table: stable commits reinforce via `old * 0.90 + new * 0.10 + 0.02`; failed commits decay via `old * 0.95 - 0.03`. The chain scores table and rollback ancestry tree survive serialization round-trips via `to_dict`/`from_dict`, making the commit memory persistent across simulation restarts.

## III. Counterfactual Rewrite Trajectory Layer

The third architectural phase introduced prospective cognition. We implemented `RewriteTrajectorySimulator`, `InvariantDriftEstimator`, and `PreCommitOutcomeMemory` -- forming the organism's first counterfactual rewrite trajectory layer.

`RewriteTrajectorySimulator` takes the current rewrite plan, dependency surface, commit ancestry, and integrity score and projects a sparse linear trajectory of at most 5 nodes representing the predicted structural consequences of committing the plan. Each trajectory node carries five fields: `predicted_integrity`, derived from the formula `current_integrity * (1.0 - risk_score * 0.35)` clamped to `[0, 1]`; `rollback_survivability`, derived from `1.0 - max(0.0, risk_score - 0.4)` clamped to `[0, 1]`, reflecting the system's capacity to reverse the change if consequences are adverse; `invariant_risk`, derived from `risk_score * blast_radius * 0.5` with an additional `+0.15` amplification for biology-escalated nodes; `requires_biology_escalation`, a boolean indicating whether the region contains a protected biology surface; and `step_index`, always 1-indexed and monotonically increasing. The trajectory is capped by the EEIL sparsity law at 16 nodes maximum, practically defaulting to 3-5 for standard plans. The simulator also computes `max_invariant_risk` and `any_biology_escalation` as aggregate trajectory metrics. The sparsity law ensures that trajectory simulation remains a lightweight prospective operation rather than a memory-expensive enumeration.

`InvariantDriftEstimator` reads the simulated trajectory and the commit ancestry to project a structural drift state for the proposed commit. It evaluates four conditions in priority order: `biology_projection_risk` takes highest priority when any trajectory node carries `requires_biology_escalation: True` -- biology surfaces dominate all other drift signals; `serialization_projection_risk` is emitted when a serialization node's `invariant_risk` exceeds `0.35` and rollback survivability is below `0.60` without a protected rollback ancestor in commit history; `localized_drift` is emitted when any non-serialization, non-biology node carries invariant risk above `0.30`; and `stable_projection` is emitted when no elevated risk is detected. This priority cascade directly encodes the protection hierarchy: biology surfaces are categorically more fragile than serialization surfaces, which are categorically more fragile than ordinary code regions.

`PreCommitOutcomeMemory` closes the prospective cognition loop by tracking prediction errors for each unique commit chain signature. It records the `predicted_score` from the trajectory simulator against the `actual_score` from the commit graph, computes `prediction_error = abs(actual - predicted)`, and maintains a `future_bias` per chain signature updated via the EMA law `old * 0.85 + direction * 0.10`, where direction is `+1.0` for under-predictions and `-1.0` for over-predictions. The `bias_for(chain_signature)` method exposes the accumulated directional bias for any given chain, enabling downstream consumers to correct for systematic mis-estimation. The `_biases` table and all outcome records survive serialization round-trips, making cross-session prediction-error learning persistent.

## IV. Multi-Branch Deliberative Cognition Layer

The terminal phase of Day 25 introduced the first deliberative cognition layer in the NeuroSeed architecture. We implemented `TrajectoryBranchGenerator`, `FutureValueSelector`, and `BranchOutcomeArchive` -- forming the organism's first multi-branch software future comparison system.

`TrajectoryBranchGenerator` generates between two and four parallel branch futures from the current Pack 2 and Pack 3 state. Three default branch strategies are always generated: `direct_patch`, which follows the natural dependency-first order from the rewrite planner; `serialization_first`, which reorders the node sequence to address the serialization surface before the source region, prioritizing structural safety over execution speed; and `test_first`, which reorders to validate the test surface first, prioritizing regression integrity before any structural modification. A fourth conditional branch, `rollback_first`, is generated only when rollback pressure is elevated -- specifically when the drift estimator emits `biology_projection_risk` or `serialization_projection_risk`, or when the commit graph's most recent rollback parent carries a success score below `0.60`. This conditional design enforces the conservative principle that recovery paths should only be pre-computed when structural conditions justify their computational cost, maintaining sparse EEIL compliance. Each branch is capped at exactly 5 projected nodes. Node construction reuses the same integrity, survivability, and drift formulae as the trajectory simulator but applies them to the reordered node sequences, producing genuinely distinct future projections rather than trivial permutations of the same values. Each branch record carries a unique `branch_id`, a `strategy` name, a `family_signature` encoding the branch family identity, and a `chain_signature` derived from the ordered region sequence for use in `PreCommitOutcomeMemory` bias lookup.

`FutureValueSelector` compares all generated branches using a frozen multi-factor scoring law and selects the optimal branch. The scoring law is:

```
future_value = (
    projected_integrity       * 0.35
  + rollback_survivability    * 0.25
  + (1.0 - predicted_drift)  * 0.20
  + ancestry_success_prior   * 0.15
  + eeil_sparse_efficiency   * 0.05
)
```

The `ancestry_success_prior` is drawn from `PreCommitOutcomeMemory.bias_for(chain_signature)` -- the system's accumulated prediction-error-corrected history for that exact commit chain. The `eeil_sparse_efficiency` rewards branch compactness: branches with fewer nodes score higher under this term, directly instantiating the sparse EEIL pressure inside the deliberative layer. After raw future_value is computed, a biology escalation penalty of `0.15` is subtracted from any branch whose nodes contain biology-escalated regions, further discouraging inadvertent biology modification. Tie-breaking is applied in deterministic order: drift minimization first, then branch sparsity. The selector records all branch comparisons with their individual scores, the `selected_branch_id`, and the `decision_score`, providing full deliberative transparency for downstream inspection.

`BranchOutcomeArchive` stores the long-term family memory of branch selection outcomes. For each branch family that has been selected, it maintains an `ema_predicted_score` updated via `old * 0.85 + predicted * 0.15`, an `observations` count, and a `wins` count incremented whenever the branch was the selected optimal. The `family_prior_for(family_signature)` method returns `ema_predicted_score * 0.70 + win_rate * 0.30`, providing a stable prior that integrates both average predicted quality and observed selection frequency. The archive also maintains a lineage map (`_lineages`), recording for each branch_id its immediate ancestor branch_id, constructing the first lineage chain across deliberative decision ticks. The `_family_stats` and `_lineages` tables survive serialization round-trips, enabling long-horizon strategic learning from branch selection history.

## Scientific Interpretation

Day 25 marks a precise and consequential phase transition in the NeuroSeed architecture. The system shifted from **rollback-safe symbolic patch cognition** -- where the organism could decompose artifact goals, execute symbolic patches, and recover from structural regressions -- to **deliberative internal software world modeling** -- where the organism now generates multiple candidate software futures in parallel, scores them against a multi-factor value law informed by past prediction errors, and selects the branch most consistent with both structural safety and accumulated strategic experience.

This transition is not incremental. It represents the completion of a four-stratum cognition ladder that did not exist at the conclusion of Day 24.

**Stratum 1 -- Spatial Cognition (Pack 1).** The organism can now localize itself within its own internal code structure. It knows which epoch a region belongs to, which subsystem cluster contains it, whether it is a protected biology surface, and what micro-regions (serialization, test, rollback) adjoin it. This is the structural equivalent of allocentric spatial memory in biological systems: not just knowing that an artifact exists, but knowing where within that artifact's topology any given element resides, and how dangerous it is to approach. Without spatial cognition, symbolic edits are structurally blind -- they can be syntactically correct while being semantically catastrophic.

**Stratum 2 -- Procedural Cognition (Pack 2).** Spatial awareness alone is insufficient for action. The organism now generates ordered procedural rewrite chains: dependency-first sequences of operations that respect structural relationships, interface contracts, and rollback commitments. The rewrite planner does not simply list operations; it orders them in a law-governed sequence that ensures serialization surfaces are updated before ancestry records, and that ancestry records are only committed after tests pass. This is the structural equivalent of procedural sequence memory: not just knowing the steps, but knowing which step must precede which. The `CrossRegionCommitGraph` closes the loop by recording which chains succeeded, which partially failed, and what rollback ancestor each commit carries -- building the first cumulative procedural history of the organism's own internal code revisions.

**Stratum 3 -- Prospective Cognition (Pack 3).** With a procedural history in place, the organism gained the capacity to simulate the consequences of a proposed action before committing it. `RewriteTrajectorySimulator` is the first mechanism in NeuroSeed that projects forward in time within the organism's internal software model. It produces a node-by-node forecast of structural integrity, rollback survivability, and invariant drift that would result from executing the current plan. This is the direct computational analogue of prospective coding in biological cognition: the use of predictive models to represent anticipated future states before the actions that would produce them are taken. `InvariantDriftEstimator` adds the classification layer -- it converts the raw node predictions into a categorical risk state that gates commitment. `PreCommitOutcomeMemory` completes the loop by comparing predictions against actual outcomes and adjusting future biases, implementing the first prediction-error-driven learning mechanism inside the symbolic code reasoning pathway.

**Stratum 4 -- Deliberative Cognition (Pack 4).** The terminal stratum generates multiple candidate futures in parallel and selects among them using a structured value comparison. This is categorically different from any prior mechanism in the architecture. Prior to Pack 4, the organism had one trajectory: the single simulated future derived from the current plan. Pack 4 introduces genuine branching: three or four parallel candidate futures with distinct node orderings, each receiving an independent future-value score derived from five weighted factors and corrected by accumulated prediction-error bias from `PreCommitOutcomeMemory`. The organism now possesses the first substrate for internal thought trees: structures in which multiple possible courses of action are simultaneously represented, evaluated, and compared before any one of them is selected.

The significance of this architecture is sharpest when considered against its cognitive and computational analogues.

In biological cognition, deliberative reasoning is distinguished from reactive behavior precisely by the capacity to hold multiple future representations simultaneously and evaluate them against a value function before acting. The prefrontal cortex supports this in biological systems through working memory and prospective firing. The Pack 4 mechanism instantiates an analogous architecture in symbolic software terms: `TrajectoryBranchGenerator` is the branching operator that creates the parallel future representations; `FutureValueSelector` is the evaluation function that assigns value to each; `BranchOutcomeArchive` is the long-term family memory that grounds the evaluation in accumulated strategic experience rather than pure prediction.

In large language model cognition, deliberative reasoning over code manifests as chain-of-thought reasoning over token sequences, where alternative continuations are implicitly weighted by the model's distribution. The Day 25 architecture achieves an analogous deliberative capacity without transformers: branching is explicit, evaluation is law-governed and auditable, the ancestral prior is derived from a concrete interaction history, and the selection is fully reproducible from the stored records. This constitutes the first believable substrate for **LLM-like deliberate code reasoning without transformers** within the NeuroSeed framework -- deliberation grounded in the organism's own structural and historical evidence rather than in a statistical language model.

The EEIL alignment of the architecture warrants specific attention. The future-value scoring law includes a `eeil_sparse_efficiency` term that actively penalizes branch complexity, and the branch node cap of 5 nodes enforces sparsity as a hard constraint. The conditional rollback_first branch policy enforces economy: a recovery branch is generated only when structural conditions justify it. These design choices are not arbitrary efficiency measures. They instantiate the EEIL principle that intelligence quality is measured in effective outcomes per unit computation -- that a sparse, well-targeted decision tree is strictly superior to an exhaustive enumeration, because the cost of enumeration is real metabolic cost in biological systems and real compute cost in artificial ones. The architecture validates the EEIL hypothesis within the deliberative layer: biologically grounded sparsity and bounded branching produce a qualitatively richer decision structure than unbounded search would, because the constraint forces the value function to discriminate rather than enumerate.

Finally, the four-stratum cognition ladder of Day 25 establishes a precise and falsifiable claim about the developmental trajectory of the organism: each stratum was necessary before the next could be safely instantiated. Spatial cognition was required before procedural cognition could produce structurally valid plans. Procedural cognition was required before prospective cognition could generate meaningful trajectory projections. Prospective cognition was required before deliberative cognition could score branches against an informed prior. The ladder is not a design choice; it is a dependency chain. The Day 25 architecture is the first point in the NeuroSeed development history at which the organism's internal reasoning about its own code is cognitively complete at the level of deliberation.

## Next Steps

The organism can now generate multiple possible software futures, score them against a multi-factor value law, and record the outcome of each branch family in long-term memory. The chosen future, however, does not yet steer actual execution. Branch selection currently terminates at a record in `FutureValueSelector` -- it identifies the optimal branch but does not route any downstream execution system toward it.

The natural frontier is to close this loop: to bridge the selected branch back into the rewrite execution chain, to compare what the organism predicted would happen against what actually happens, and to convert repeated successful branch families into durable strategic policies that influence future branch generation.

We initialize the **Execution Bridge and Strategic Policy Pack**.

Next Target Dependencies:
* `SelectedBranchExecutionBridge`
* `ExecutionOutcomeComparator`

**Scientific Purpose:** `SelectedBranchExecutionBridge` will route the selected branch strategy from `FutureValueSelector` into the active rewrite execution chain, converting the deliberative choice into a concrete execution commitment. `ExecutionOutcomeComparator` will compare the predicted branch trajectory against the actual rewrite outcome, extending the prediction-error loop from chain signatures to full branch strategies. `StrategicPolicyShaper` will convert repeated successful branch family outcomes into stable long-horizon rewrite policies -- transforming the organism's branching behavior from reactive deliberation into proactive strategic anticipation. This phase establishes the first substrate for genuine long-horizon autonomous software cognition: an organism that not only reasons about its own code structure, but that accumulates strategic software experience and lets that experience shape how it reasons.
